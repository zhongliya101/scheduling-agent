## 1. 背景与问题

现有门店排班通常以区域为单位：水产区、肉类区、果蔬区、收银区、补货区分别安排员工。该方式可以保障专业岗位稳定，但在实际经营中暴露出三个问题：

| 编号 | 问题 | 描述 |
| --- | --- | --- |
| P-01 | 系统推荐后仍需要大量人工调整 | 店长经常根据"谁是师傅""谁不能离开本区域""谁可以跨区帮忙"等经验改班，说明系统没有充分理解门店真实用人规则 |
| P-02 | 区域固定导致低峰人力利用不足 | 某些区域低峰有人闲置时，其他区域可能出现补货、称重、收银、打包等通用任务缺口 |
| P-03 | 全员混排不可接受 | 杀鱼、切肉、熟食加工等岗位依赖专业技能和稳定人员，不能简单按照"谁有空就排谁"分配 |

因此，本 Demo 采用**半混班**方案：专业岗位固定保障，通用岗位跨区补位。

---

## 2. 版本结论

本 Demo 只验证一个核心命题：

> 在生鲜商超门店中，系统先稳定锁定水产、肉类等专业区域的关键师傅，再使用全店混排池补齐高峰和通用任务缺口，从而减少店长手动改班，并提升通用人力利用率。

本版本聚焦单门店半混班智能排班场景，核心目标是验证 **Agent 能基于历史经营数据、高峰/低峰规律、节假日、天气等因素综合估算用工需求，并在半混班约束下生成最优或近似最优排班方案**。

研发实现应优先保证三件事：

1. **需求计算可信**：能说明为什么某天某时段需要更多或更少人。
2. **排班结果可用**：专业岗稳定，通用人力能跨区补位。
3. **Agent 可解释**：能解释预测依据、排班取舍、不可调原因和替代方案。

## 3. 产品定位

### 3.1 产品一句话

智慧排班 Agent 半混班 Demo 是一个面向生鲜商超门店的排班工作台：Agent 基于历史销售、客流、订单、节假日、天气等因素计算未来 7 天分时段用工需求，再在专业岗稳定、混排池补位、区域保底和工时约束下生成半混班班表，并解释每一次需求判断、固定、混排和不可调整原因。

### 3.2 Demo 成功标准

| 编号 | 标准 | 说明 |
| --- | --- | --- |
| S-01 | 能生成 1 家门店 7 天半混班班表 | 完整输出一周排班 |
| S-02 | 专业固定岗优先由专业人员覆盖 | 杀鱼、切肉等岗位由 S/A 级师傅承担 |
| S-03 | 高峰缺口可优先从混排池补位 | 不轻易打乱专业固定岗 |
| S-04 | 系统能解释为什么固定、为什么混排、为什么不可调 | 每个决策都有可追溯的解释 |
| S-05 | 店长人工修改能被记录、归因和统计 | 记录干预率和高频原因 |
| S-06 | 5 分钟内可完整演示闭环 | 从需求计算→生成班表→发现缺口→Agent 推荐→解释不可调→记录干预 |

## 5. 关键概念

| 概念 | 定义 | 示例 |
| --- | --- | --- |
| 专业固定岗 | 需要专业资质或高熟练度，仅由具备资质员工独立承担的岗位 | 杀鱼、切肉 |
| 区域专业人士 | 长期稳定服务某区域，具备专业固定岗能力的核心员工 | 水产师傅、肉类师傅 |
| 区域员工 | 主要在某区域工作，可承担本区域基础任务，部分可跨区支援 | 小林、小赵 |
| 全店混排池 | 可跨区域承担通用任务的员工集合 | 补货、称重、打包、基础收银支援 |
| 混排任务 | 不强依赖专业固定岗能力，可由具备基础技能的员工承担的任务 | 补货、陈列、打包、称重 |
| 保护时段 | 专业人士必须优先留在本区域的时段 | 水产早市、肉类晚高峰 |
| 可支援时段 | 本区域低峰且不影响区域保底时，可释放人员支援其他区域的时段 | 14:00-16:00 |
| 区域保底人数 | 每个区域关键时段必须保留的最低人数 | 水产早高峰至少 2 人 |
| 人工干预 | 店长对系统推荐班表进行的新增、删除、替换、调时、改区域操作 | 把小唐从果蔬改到收银 |

## 7. Demo 业务流程

### 7.1 主流程

1. 店长进入半混班排班工作台。
2. 系统加载样例门店、区域、岗位、员工、技能、需求数据。
3. 店长点击"生成下周半混班班表"。
4. 系统先锁定专业固定岗和保护时段。
5. 系统校验各区域保底人数。
6. 系统生成全店混排池。
7. 系统按高峰缺口和任务优先级进行混排补位。
8. 系统输出周班表、区域覆盖、风险提示、KPI 和解释。
9. 店长查看缺口或风险，可点击 Agent 提问。
10. 店长进行人工修改时，系统记录修改项和原因。
11. 系统更新干预率、风险和相关解释。

### 7.2 核心演示路径

| 步骤 | 用户操作 | 系统反馈 |
| --- | --- | --- |
| 1 | 点击"生成下周半混班班表" | 生成 7 天班表，展示专业固定岗和混排岗 |
| 2 | 查看周五晚高峰 | 展示果蔬/收银缺口和可补位候选人 |
| 3 | 向 Agent 提问"周五晚高峰果蔬缺人，谁能支援" | 推荐小唐、小马等候选人，并解释推荐原因 |
| 4 | 向 Agent 提问"能不能把老张调去收银" | 说明老张为肉类切肉师傅，保护时段不能抽调 |
| 5 | 手动修改一条排班 | 弹出修改原因，记录干预并更新人工干预率 |
| 6 | 查看 KPI | 展示专业岗覆盖率、区域保底达成率、混排利用率、人工干预率 |

## 9. 业务规则

### 9.1 规则优先级

| 优先级 | 规则类型 | 处理方式 |
| --- | --- | --- |
| 1 | 专业岗资质 | 无资质员工不得独立承担专业固定岗 |
| 2 | 保护时段 | 保护时段内专业人士不得被跨区抽调 |
| 3 | 区域保底 | 核心区域关键时段必须满足最低保底人数，无法满足则标记风险 |
| 4 | 员工可用性 | 不在可用时段内的员工不推荐排班 |
| 5 | 周工时上限 | 超出上限标记风险，不作为首选候选人 |
| 6 | 混排技能匹配 | 混排员工必须具备目标任务最低技能 |
| 7 | 区域熟悉度 | 曾支援过该区域者优先 |
| 8 | 工时均衡 | 当前周工时较低者优先 |
| 9 | 员工偏好 | 满足偏好者优先 |

### 9.2 硬规则

| 编号 | 规则 | 说明 | 失败处理 |
| --- | --- | --- | --- |
| R-001 | 专业岗必须由专业人员覆盖 | 杀鱼、切肉等岗位要求 S/A 级专业技能 | 标记严重风险，不推荐普通员工 |
| R-002 | 核心区域保留 2-3 名专业/区域人员 | 防止区域被混排抽空 | 标记区域保底不足 |
| R-003 | 保护时段不得抽调专业人士 | 高峰和关键加工时段优先留守本区域 | Agent 解释不可调 |
| R-004 | 混排员工必须具备基础技能 | 例如收银必须有收银权限，称重/补货至少 B 级 | 不进入候选列表 |
| R-005 | 周工时不能超过上限 | 每名员工配置周工时上限 | 标记超工时风险，默认不推荐 |

### 9.3 软规则

| 编号 | 规则 | 说明 | 排序影响 |
| --- | --- | --- | --- |
| R-101 | 优先使用混排池补缺口 | 不轻易打乱专业固定岗 | 提高混排员工优先级 |
| R-102 | 工时尽量均衡 | 避免个别员工过多或过少 | 当前工时低者优先 |
| R-103 | 区域熟悉度优先 | 曾支援目标区域者更可信 | 熟悉度高者优先 |
| R-104 | 高峰关键任务优先 | 高峰先保收银、称重、加工、补货 | 高峰任务先排 |
| R-105 | 减少人工干预 | 记录高频修改原因，后续用于规则优化 | 本 Demo 只统计，不自动学习 |

## 11. 排班生成逻辑

### 11.1 输入

排班生成至少需要以下输入：

- 历史销售、客流、订单、排班数据
- 节假日、周末、天气、促销等影响因子
- 门店区域配置
- 区域任务和专业岗配置
- 保护时段配置
- 区域保底人数配置
- 员工列表
- 员工技能等级
- 员工类型和主区域
- 员工周工时上限
- 7 天分时段需求数据

### 11.2 输出

排班生成至少输出以下结果：

- 7 天分时段、分区域、分任务的需求计算结果
- 每个需求结果的需求因素、需求强度和置信度
- 7 天半混班班表
- 每条班次的员工、日期、时段、区域、任务、标签
- 专业岗覆盖率
- 区域保底达成率
- 混排利用率
- 高峰缺口数
- 风险列表
- 每条关键安排的解释
- 混排候选人推荐列表

### 11.3 生成步骤

1. 读取配置和需求数据。
2. 读取历史销售、客流、订单、排班、节假日、天气、促销等数据。
3. Agent/需求计算模块识别高峰、低峰、节假日、天气和促销影响。
4. 生成 7 天区域/任务级需求：需求人数、需求强度、需求因素、置信度。
5. 识别所有专业固定岗任务。
6. 按专业技能 S/A、主区域、保护时段优先锁定专业师傅。
7. 计算各区域保护时段已排人数和保底缺口。
8. 将非保护时段、非专业任务、高峰缺口加入待补位任务池。
9. 生成混排候选人池。
10. 对待补位任务按优先级排序：高峰 > 普通 > 低峰；收银/称重/加工/补货优先于清洁等低优任务。
11. 对每个任务筛选候选人：技能满足、可用、未超工时、不影响主区域保底。
12. 对候选人排序：技能等级 > 区域熟悉度 > 剩余工时 > 员工偏好。
13. 分配最优候选人并更新其周工时。
14. 对未满足需求生成缺口和风险。
15. 生成 KPI、预测解释、排班解释和可视化数据。

### 11.4 候选人评分公式

```
候选人得分 =
  技能匹配分 × 40
  + 区域熟悉度分 × 20
  + 剩余工时分 × 20
  + 偏好匹配分 × 10
  + 主区域不受影响分 × 10
  - 风险扣分
```

| 评分项 | 取值说明 |
| --- | --- |
| 技能匹配分 | S=1.0, A=0.9, B=0.7, C=0 |
| 区域熟悉度分 | 主区域=1.0, 曾支援=0.8, 未支援但可做=0.5 |
| 剩余工时分 | 剩余工时越多分越高 |
| 偏好匹配分 | 满足偏好=1, 不满足=0 |
| 主区域不受影响分 | 不影响主区域保底=1, 否则=0 |
| 风险扣分 | 超工时、技能弱、跨区频率高等分别扣分 |

### 11.5 风险等级

| 等级 | 颜色 | 含义 | 示例 |
| --- | --- | --- | --- |
| 严重 | 红色 | 影响专业岗或关键服务，不建议通过 | 杀鱼岗位无人覆盖 |
| 警告 | 黄色 | 可演示但需要店长确认 | 混排员工技能仅 B 级 |
| 提示 | 灰色/蓝色 | 信息提示，不影响排班 | 某员工本周跨区 3 次 |

## 13. Agent 需求

### 13.1 Agent 定位

Demo 阶段 Agent 是核心排班编排入口，负责理解历史数据、影响因子、半混班规则和用户意图，输出需求计算、排班建议、解释和调整方案。Agent 不直接绕过规则修改正式班表，所有调整建议都需要用户点击确认或手动应用。

### 13.2 支持意图

| 意图编码 | 用户示例 | 系统响应要求 |
| --- | --- | --- |
| INTENT_GENERATE | 生成下周半混班班表 | 触发或说明生成逻辑 |
| INTENT_EXPLAIN_DEMAND | 为什么周五晚高峰果蔬需要 3 个人 | 解释历史客流、订单、节假日、天气等需求因素 |
| INTENT_EXPLAIN_FIXED | 为什么老王一直在水产区 | 解释专业技能、保护时段、区域保底 |
| INTENT_RECOMMEND_SUPPORT | 周五晚高峰果蔬缺人，谁能支援 | 返回候选人排序和推荐理由 |
| INTENT_EXPLAIN_BLOCKED | 能不能把老张调去收银 | 返回不可调原因和替代方案 |
| INTENT_LIST_RISKS | 哪些地方需要店长确认 | 列出风险和需确认项 |
| INTENT_REDUCE_INTERVENTION | 怎么减少人工干预 | 汇总干预原因并建议调整规则 |

### 13.3 Agent 返回格式

Agent 返回应包含三类信息：

1. **结论**：能不能做、推荐谁、风险是什么。
2. **原因**：对应历史数据、峰谷规律、节假日、天气、技能、区域、时段、工时、规则。
3. **下一步**：可点击应用、查看详情、选择其他候选人或保留原班表。

**推荐候选人示例：**

```
建议优先安排小唐支援周五 17:00-19:00 果蔬区。

原因：
1. 小唐具备果蔬 A、补货 A、打包 A 技能，可独立完成该时段通用任务。
2. 小唐当前周工时未超过 32 小时上限。
3. 该安排不影响水产、肉类保护时段的专业岗保底。

备选：
- 小马：可做称重和打包，但不建议独立处理高峰补货。
- 小何：具备补货能力，但当天已接近周工时上限。
```

**不可调解释示例：**

```
不建议把老张调去收银。

原因：
1. 老张是肉类切肉 S 级师傅，当前时段属于肉类晚高峰保护时段。
2. 肉类区该时段需要保障切肉和分割任务，抽调后会导致专业岗覆盖不足。
3. 收银缺口可优先从混排池补位，例如小唐或小郑。
```

## 14. KPI 与计算口径

| 指标 | 计算公式 | Demo 目标 |
| --- | --- | --- |
| 需求解释覆盖率 | 可解释需求时段数 / 关键需求时段数 | 覆盖高峰、低峰、节假日、天气四类 |
| 专业岗覆盖率 | 已由 S/A 专业人员覆盖的专业岗需求数 / 专业岗总需求数 | 100% |
| 区域保底达成率 | 达成保底的核心区域关键时段数 / 核心区域关键时段总数 | ≥ 90% |
| 混排利用率 | 混排支援工时 / 总排班工时 | 20%-35% |
| 高峰缺口数 | 高峰时段未满足需求人数之和 | 比纯区域固定排班少 |
| 人工干预率 | 被人工修改的排班项数 / 系统生成排班项数 | 可记录、可解释 |
| Agent 解释覆盖率 | 可被 Agent 解释的关键排班/风险数 / 关键排班/风险总数 | 覆盖固定岗、混排、不可调三类 |

## 16. 非功能需求

| 类别 | 要求 |
| --- | --- |
| 性能 | 点击生成后可在可接受等待时间内返回需求计算和 Demo 班表；Agent 响应需有加载态和失败兜底 |
| 可用性 | Demo 主流程可使用本地样例数据；如 LLM 不可用，应提供基础规则兜底结果 |
| 可解释性 | 关键需求计算、排班、风险、不可调建议必须有文字解释 |
| 可维护性 | 样例数据、区域、员工、规则建议用 JSON/常量配置，避免硬编码散落 |
| 可演示性 | 页面默认加载完整样例数据，支持零录入演示 |
| 容错 | 无候选人时必须展示原因和处理建议 |
| 安全 | Demo 不展示真实个人敏感信息，员工使用样例姓名 |

## 17. 验收标准

### 17.1 功能验收

| 验收项 | 标准 |
| --- | --- |
| 需求计算 | 能基于历史销售、客流、订单、节假日、天气等样例数据生成区域/任务级需求 |
| 需求解释 | 能解释某时段需求高低的主要因素 |
| 生成半混班班表 | 可生成 1 家门店 7 天班表 |
| 专业岗锁定 | 杀鱼、切肉等岗位优先由老王/老陈/老张/老刘等专业师傅覆盖 |
| 区域保底 | 水产、肉类等核心区域保护时段可展示是否达到保底 |
| 混排池 | 可展示全店混排池及每名员工可支援区域 |
| 混排补位 | 周五晚高峰果蔬/收银缺口可推荐候选人 |
| 不可调解释 | 尝试抽调专业师傅时能解释原因并给替代方案 |
| 干预记录 | 修改班次后必须记录修改项和原因 |
| KPI 展示 | 展示专业岗覆盖率、区域保底达成率、混排利用率、人工干预率、高峰缺口数 |
| Agent 解释 | 支持固定岗、混排、不可调、风险、减少干预五类问题 |

### 17.2 演示验收

5 分钟内必须完成以下演示：

1. 打开半混班排班工作台。
2. 点击生成下周半混班班表。
3. 展示 Agent 对历史数据、高峰/低峰、节假日、天气等因素的需求计算结果。
4. 展示专业岗锁定和混排标签。
5. 定位周五晚高峰果蔬/收银缺口。
6. Agent 推荐支援人员。
7. Agent 解释为什么不能抽调老张。
8. 手动修改一条排班并选择原因。
9. 展示人工干预率和干预原因。

## 18. 边界场景

| 场景 | 系统处理 |
| --- | --- |
| 历史数据不足 | 使用样例基线和规则兜底，并提示置信度较低 |
| 天气/节假日数据缺失 | 使用历史同星期/同时间段基线计算，并提示缺失因素 |
| 专业师傅不足 | 标记严重风险，说明缺少 S/A 专业人员 |
| 混排池无人可用 | 展示无候选原因：技能不满足、超工时、影响主区域保底等 |
| 候选人技能较弱 | 可推荐但标记黄色提醒 |
| 调走专业师傅会破坏保底 | 拒绝推荐，并给混排池替代方案 |
| 手动修改导致超工时 | 标记风险并要求填写原因 |
| 同一员工跨区过多 | 标记提示，建议减少跨区频次 |
| 用户提问超出 Demo 范围 | Agent 说明当前 Demo 暂不支持，并引导到支持问题 |

## 19. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 规则表达不清 | 研发实现口径不一致 | 以本 PRD 的规则优先级和验收标准为准 |
| 样例数据不足 | 演示看不出混排价值 | 必须内置周五晚高峰缺口和不可调案例 |
| LLM 输出不稳定 | 解释不可信或建议越界 | LLM 只做编排和解释，排班结果必须经过规则校验，关键指标由程序计算 |
| 需求计算依据不足 | 排班看起来像写死 | 页面展示历史数据、峰谷、节假日、天气等需求因素 |
| 页面信息过载 | 5 分钟演示难以讲清 | 主页面只保留需求洞察、KPI、区域、周班表、热力图、Agent 六块 |
| 算法结果不稳定 | 每次演示结果不一致 | Demo 使用确定性规则和固定样例数据 |
